Buyer + orders + notify — implementation notes

Decisions & deviations a reviewer should confirm

  • BackgroundTasks block the request when run through TestClient/ASGITransport. Starlette's Response.__call__ awaits background BEFORE returning from the ASGI call, and both fastapi.testclient.TestClient and httpx.ASGITransport invoke the app fully synchronously from the caller's point of view — so POST /orders in tests/smoke would otherwise block for the FULL ACK_TIMEOUT_SECONDS (120s default) before returning. Every test file and the smoke script explicitly set ACK_TIMEOUT_SECONDS=0.05-0.1 before hitting an order-creating endpoint. In a REAL uvicorn deployment the HTTP response bytes are flushed to the client immediately (buyer sees a fast response) but the server-side request coroutine still lingers for up to ACK_TIMEOUT_SECONDS — fine at pilot volume, but a real task queue (Cloud Tasks / a proper scheduler) should replace BackgroundTasks once concurrent order volume matters.
  • Ack-timeout races the response in the smoke script (and would in any single-request flow): the background watcher's sleep can already elapse before the seller-app calls POST /orders/{id}/ack, since — per the point above — the watcher runs to completion INSIDE the same await as the order-creation response when there's no other concurrent work keeping the event loop busy. scripts/e2e_smoke.py step 10 therefore observes the SMS fallback firing immediately, before the explicit ack call; this is a smoke-harness artifact, not a race in real deployment (there, other requests interleave normally). tests/test_notify.py::test_race_seller_seen_flips_mid_wait covers the real "ack lands mid-sleep" case directly against NotifyPipeline.watch_ack.
  • Fuzzy dish-name matching scores by QUERY precision, not dish recall (order_tools.fuzzy_match_dish): score = |query∩dish| / |query|, not / |dish|. Chat orders are almost always an abbreviated fragment of the menu's fuller name ("gà nướng" for "Cơm tấm gà nướng mật ong") — scoring by how much of the DISH name is covered would unfairly punish long, descriptive menu names and miss real matches. Ties (a very short/generic query matching several dishes at 1.0) break toward the shortest dish name, then dish_id, for deterministic output — not a real disambiguation, just keeps tests stable; a genuinely ambiguous one-word order should still get a low-confidence warning path in a later iteration.
  • Heuristic order-text parser only attempts a fuzzy match on a quantity-less segment if it ALREADY scores a match (order_parse_agent._looks_like_item_segment): a bare-digit/×-notation segment ("3 xíu mại đặc biệt") is always tried against the menu (so a genuine miss surfaces as a review warning), but a segment with no quantity marker AND no token overlap at all ("Chào shop") is silently skipped rather than generating a warning. Trade-off: an unmatched item mentioned without an explicit quantity would be silently dropped instead of warned about — accepted for MVP noise reduction; the import agent's OCR path doesn't have this problem because vision models are far more reliable at "is this text a dish line" than a regex heuristic.
  • Order-parse draft never becomes a real order — POST /orders/parse-text mirrors the import agent's review-gate pattern (ARCH §3.1): it returns {"items","customer","warnings","confidence"} for the seller app to correct, then the existing POST /orders finalizes. No wiring exists (yet) for the seller app UI itself — out of scope for this slice per the task list (agent + endpoint only).
  • Group-order member re-submission REPLACES, not appends (GroupOrderStore.add_member_items): a member editing their pick (add a drink, fix a quantity) re-POSTs their FULL list under their name; it isn't a running ledger. Two different people typing the same name would collide (last write wins) — acceptable for a same-office, no-login flow where names are already the identity key (ARCH §5.4 "không login cho buyer").
  • Group-order split pays out ACTUAL consumption, not an equal division — ARCH §3.3 says "chia tiền hiển thị từng người", which this reads as "show what each person themselves ordered", not "divide the bill by headcount". This is why the split always sums EXACTLY to the order total with no rounding remainder (asserted in GroupOrderStore.close) — there's nothing to round, each member's amount IS their own subtotal. The payer (closer) is required to be a member (they must have ordered ≥1 item) since ARCH frames them as "người trả hộ", i.e. one of the group who fronts the money.
  • VietQR payload is INTERFACE ONLY (bank/account are None) — real bank-deeplink QR generation is infra/vietqr.py, an explicitly later phase per ENGINE-SPEC §3/ARCH §5.4. The shape (payee, amount, note) is fixed now so the buyer UI and that future module already agree on a contract.
  • Server is the price + sold-out authority for every order path (solo checkout AND group members): client-sent price/name in items is ignored; the dish is re-resolved from the shop's own stored menu doc, and a dish flagged sold_out 409s the request. This also catches the race where a buyer's cached page still shows a dish that just sold out.
  • Renderer re-renders the whole surface on every mutation (add-to-cart, reorder-card inject, sold-out patch, order-status update) instead of patching the DOM incrementally. Fine at MVP page size (buyer bundle is ~42KB total, comfortably under the 150KB target) — a virtual-DOM-style diff would be the first thing to add if a menu grows large enough to make full rebuilds visibly slow on a 4G phone.
  • /t/{slug} and /g/{gid} inject a bootstrap <script> into the COMMITTED static HTML rather than templating it: the server reads buyer/index.html/buyer/group.html off disk, string-inserts window.__TIEMQUEN__ = {...} right after <head>, and returns it as HTMLResponse. The SAME files are also served byte-for-byte (no injection) via the existing /buyer static mount, with the buyer JS falling back to a ?slug= query param in that case — used by the dev/test path GET /buyer/index.html?slug=... and by hand-testing without hitting /t/ at all.
  • Context-rule batch classification is a substring heuristic (classifyBatch: folded batch id matches office|van-phong|vp → office variant, else table) — ENGINE-SPEC §9 doesn't specify the batch-id naming scheme, and there's no batch registry yet to look up a shop-defined kind. The RAW ?b= value is still stored verbatim as the order's batch_id (flyer analytics per the task spec) regardless of how it's classified for variant selection.
  • Sold-out "fresh patch" on the buyer page re-fetches GET /api/shops/{slug}/menu (the seller-review endpoint) rather than a new dedicated public endpoint — it already returns exactly sold_out/almost_out per dish, is not slug-registry-gated any differently, and adding a bespoke endpoint would just be that same data reshaped. No cache-control header is set on it yet (todo: short Cache-Control: max-age=15-ish once this is under real load, per ENGINE-SPEC §9's "cache-control ngắn").

Open questions

  • Should the ack-timeout watcher move off FastAPI BackgroundTasks to a real scheduler before pilot launch, given the "ties up the request coroutine for up to 120s" behavior noted above? Fine at 5-shop pilot scale; revisit if order volume/concurrency grows.
  • infra/orders.py's OrderStore.list_by_shop is a full collection scan (no shop-id index) — unused by any route in this slice (no seller dashboard yet) but will need an index once that lands.
  • Real-mode order_parse_agent._real_parse (toolable loop over the same add_item/set_customer/finish tools) is written but, like the composer's real mode, has never been exercised against the live Gemini API.
