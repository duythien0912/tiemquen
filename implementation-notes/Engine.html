Engine (compose) — implementation notes

Decisions & deviations a reviewer should confirm

  • Endpoint prefix: task said POST /shops/{slug}/compose|patch; implemented as POST /api/shops/{slug}/compose and /api/shops/{slug}/patch to match the existing /api/shops CRUD prefix (bare /<slug> is reserved for the public buyer URL space per infra/publish.py). Also added GET /api/shops/{slug}/composed/{variant} so the dev buyer page can fetch the cached A2UI.
  • Patch coalescing: spec says patches are appended; a repeated patch on the SAME data path replaces the previous trailing patch message instead of appending again, so a dish toggled sold-out N times doesn't grow the cached file forever. Patches on different paths still stack in order. (compose/cache.py::patch_data)
  • Validator repair vs raise policy (SPEC §1 only lists the repairs): repairable → fix + warning (missing version, inferable action key, missing createSurface/surfaceId/root, bare childIds list/pointer, raw literals wrapped into leaf shapes, dangling child refs pruned, malformed events/props dropped). Unrepairable → A2UIValidationError (uninferable action key, missing/empty components, root id absent, bad updateDataModel path, component type not in catalog when a catalog is passed) — real mode catches this to drive the one correction-prompt retry.
  • Mock composer is held to zero warnings: template output runs through the same validator as LLM output and raises if any repair fired — keeps the deterministic path honest and the cache guaranteed-clean.
  • DishCard component ids are section-scoped (card_<section_id>_<dish_id>): the same dish can appear in two sections (combo trưa reuses regular dishes) and flat-wire ids must be unique. The dish identity travels in the onPress event context (dishId) and in the /prices|/soldout|/almostout binding paths, not in the component id.
  • /patch also syncs the shop doc (menu.dishes[*].sold_out/almost_out/price) when the patch path is recognizably /soldout|/almostout|/prices/<dish_id>, so the next structural recompose starts from fresh flags instead of silently reverting them. Unrecognized paths patch the cache only.
  • GroupOrderButton exists in every variant — office batch places it right under the hero (promoted, ARCH §5.3), table batch places it after the sections. Client context rules can still re-promote without recompose.
  • Racing: race_compose() is async first-valid-wins with loser cancellation; race_sync() wraps a blocking compose call thread-per-racer (google-genai calls in composer.py are sync). Invalid results (exceptions) are skipped until a valid one lands; all-fail raises RaceError with the individual errors. Only wired into real mode, gated by RACE_ENABLED/RACE_COUNT.
  • Theme seed roles are derived, not positional: bg = lightest seed (lightened 65% toward white), text = darkest seed, accent = most saturated of the two middle seeds; success/warn are fixed hues tinted 12% toward the accent. Every pair in TEXT_PAIRS is force-nudged to WCAG ≥ 4.5 (deterministic 15% steps toward black/white with a pole fallback), so even degenerate seeds (all-black, all-white, mid-gray) yield a compliant palette.
  • Variant naming: <batch>-<daypart> = office|table × lunch|regular → 4 cached files. Lunch reordering is a heuristic on section titles (accent-folded match of "combo"/"trưa"); menus with no such section produce a lunch variant identical in structure to regular — still cached separately as spec'd.
  • data/composed/ added to .gitignore — it is runtime cache output, same class as data/media/.
