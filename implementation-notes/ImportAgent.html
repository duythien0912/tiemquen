Import agent — implementation notes

Decisions & deviations a reviewer should confirm

  • Endpoint paths: task said POST /import, GET /shops/{slug}/menu, PATCH /shops/{slug}/menu; implemented as /api/import and /api/shops/{slug}/menu to match the existing /api/shops prefix used by every other route in server.py (same reasoning as Engine's notes: bare /<slug> is reserved for the public buyer URL space).
  • POST /api/import dispatch is one handler, not three routes: it inspects the request's Content-Type — multipart/form-data with a screenshot field → OCR path; otherwise a raw JSON body with url | text | fixture. Chose this over separate endpoints because FastAPI can't cleanly mix a File/Form parameter set and a JSON Body parameter set on one route, and a single /import surface matches the task text more literally than three sub-routes would.
  • {"fixture": "<name>"} JSON form is new, not in the task text: added so scripts/e2e_smoke.py and manual/demo testing can drive the whole import → review → patch → recompose flow over real HTTP without needing actual screenshot bytes on disk or a GEMINI_API_KEY. It replays any data/fixtures/<name>.json tool-call trace (same shape as grab_screenshot_toolcalls.json) through toolable.Toolable.replay — the identical assembly path real screenshots use in MOCK mode.
  • Import does NOT persist a shop or call the image rehost: /api/import returns only the review envelope ({menu, warnings, confidence}, ENGINE-SPEC §5); the seller review step is expected to follow with the existing POST /api/shops to finalize. Image rehost (infra/media.py) is built and unit-tested standalone but deliberately NOT auto-invoked from /api/import: it makes real network calls per dish image, which would make the import endpoint's test surface network-dependent/non-deterministic. It's available for a later review/finalize step (or the seller-app "confirm import" action) to call explicitly.
  • URL import has no automatic re-attempt as OCR — a failed HTML parse raises ImportFallbackToOCR which propagates all the way to the client as HTTP 422 {"error": ..., "fallback_to_ocr": true}. There's no screenshot bytes available at a URL-import call site to retry with, so "fallback to OCR" is necessarily a separate follow-up call from the seller (upload a screenshot instead), not an automatic in-process retry.
  • html_parse.py's embedded-JSON scrape shape is a generic heuristic, not a verified ShopeeFood API contract: ARCH §4.5 only confirms Grab's HTML path is blocked (07/2026); ShopeeFood's actual embedded-JSON shape is undocumented and was not available to verify against a live page in this environment. The scraper walks any parsed JSON tree for lists of ≥2 dicts carrying a name-like key (name|title|dish_name|item_name) and a price-like key (price|price_display|sale_price| salePrice|current_price), picks the largest such list, and requires a resolvable shop name (og:site_name / og:title / <title>) — every unmet condition raises ImportFallbackToOCR. This is intentionally likely to fail on the real site until someone verifies the actual markup; OCR remains the path the product depends on (ARCH §3.1: "OCR screenshot is đường CHÍNH, not fallback").
  • Raw-text import in MOCK mode has no dedicated fixture (the task only asked for one for the screenshot/Grab path). It degrades to a small regex line-parser ("Tên món - 35.000đ" shape) instead of calling a model — zero-network, but much weaker than real OCR/LLM text parsing. REAL mode reuses the exact same toolable tool-calling loop as screenshots, just with text-only contents instead of image parts.
  • Imported menu docs have no shop.theme — OCR/HTML-parse/text sources have no way to know brand colors; that's the Imagen/flyer agent's job (ENGINE-SPEC §6, later phase). Consequence: calling POST /api/shops/{slug}/compose or PATCH /api/shops/{slug}/menu on a freshly-imported shop that skipped theme assignment will raise KeyError: 'theme' inside compose_all_variants (pre-existing behavior in compose/composer.py, not changed here). scripts/e2e_smoke.py injects a fake seed-color theme after import specifically to exercise /compose — a reviewer should confirm whether compose_all_variants should tolerate a missing theme (e.g. fall back to a default palette) once the interview/imagen agent isn't guaranteed to run first.
  • PATCH /api/shops/{slug}/menu vs the existing POST /api/shops/{slug}/patch: kept as two distinct endpoints per the task/spec split — /patch is sold-out/price-only, updateDataModel-append, no recompose (ENGINE-SPEC §1); /menu PATCH is the seller-review edit set (set_price, hide_dish, add_dish, retitle_section via shared.menu_format.apply_menu_edits) and always does a full structural recompose, since add_dish/retitle_section change the component tree, not just bound data.

Open questions

  • Should rehost_dish_images be wired automatically into the import→review flow (e.g. right before POST /api/shops persists), and if so, on what timeout budget — the current default (10s per image, no image cap) could make an import with many dish photos slow synchronously?
  • The compose_all_variants missing-theme KeyError noted above: worth a default palette fallback so an imported-but-not-yet-themed shop can still be composed (even if bland) instead of 500ing?
  • html_parse.py's heuristic scraper has never been run against a real ShopeeFood page — needs verification (or an explicit decision to drop the URL path entirely and lean 100% on OCR, matching ARCH §3.1's own framing of OCR as the main path).
