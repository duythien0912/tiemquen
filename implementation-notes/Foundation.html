Foundation phase — decisions & deviations

  • Python interpreter: Homebrew python3 (3.14.5) and python3.13 on this machine are broken — pyexpat fails to load (Symbol not found: _XML_SetAllocTrackerActivationThreshold vs system libexpat), so python3 -m venv dies inside ensurepip. The venv was created with the uv-managed CPython 3.12.12 (~/.local/share/uv/python/cpython-3.12-.../bin/python3.12 -m venv .venv), which also matches SPEC §3's "Python 3.12". Reviewer note: on other machines the plain python3 -m venv .venv from the README works; nothing in the repo depends on the uv path.
  • Schema referential integrity in code, not schema: "every sections[].items dish_id must exist in menu.dishes" is not expressible in JSON Schema across branches, so shared/menu_format.validate_menu() does the jsonschema pass plus that check. Later phases should validate via validate_menu(), not raw jsonschema.validate.
  • Slug conflicts auto-suffix instead of erroring: POST /api/shops never 409s on a taken/reserved slug — SlugRegistry.register silently assigns -2, -3, ... and the response doc carries the actually-assigned slug. Duplicate shop id does 409. Chosen so onboarding (import agent, later phase) never stalls on a name collision; confirm this matches desired UX.
  • POST /api/shops with empty body seeds the demo fixture — dev-bootstrap convenience per task ("create from fixture"). If this endpoint survives to prod it should require an explicit body.
  • Order transitions: SPEC §8 lists states but not the transition graph; codified as created→seller_seen→confirmed→delivering→done, with cancelled reachable from created/seller_seen/confirmed and no_show_flagged from confirmed/delivering. Skipping states (e.g. created→confirmed if seller confirms without an explicit "seen" ack) is currently disallowed — later notify/order phase may want to relax that.
