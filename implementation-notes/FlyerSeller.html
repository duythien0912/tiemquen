Flyer generation + seller PWA — decisions & open questions

1. VietQR "known-good reference vector" = CRC standard check value, not a memorized payload

The task asks to unit-test the EMVCo payload "against a known-good reference vector". I did NOT hardcode a full remembered VietQR string as golden (a wrongly-remembered constant would bake a bug into the test as truth). Instead the tests pin (a) the universally published CRC-16/CCITT-FALSE check value crc16("123456789") == 0x29B1 (the exact CRC EMVCo mandates: poly 0x1021, init 0xFFFF, no reflect/xorout), and (b) structural TLV assertions on our own payload: field order 00,01,38,53,[54],58,[62],63; NAPAS GUID A000000727; service QRIBFTTA; BIN/account nesting; currency 704; CRC round-trip + tamper detection. Reviewer action: scan one generated QR (infra/vietqr.qr_png) with a real banking app once before pilot — that is the only true end-to-end vector.

2. Bank BIN table is a hardcoded subset

vietqr.BANK_BINS covers ~12 common banks; unknown bank names raise, but a literal 6-digit NAPAS BIN passes through. Full NAPAS list is a data-file concern for later.

3. Transfer note is ASCII-folded

Field 62-08 strips diacritics/punctuation ("Bình trả" → "Binh tra"): bank cores routinely mangle UTF-8 in transfer notes and EMV length counting is per-character; ASCII keeps chars==bytes.

4. Imagen palette → shop theme; publish ORDER matters

ENGINE-SPEC §6: theme comes from the 4 imagen seed colors. POST /api/shops/{slug}/hero persists the palette into shop.theme.seed_colors — but only when the shop has no theme yet (never overwrites a seller-picked theme). Because compose/recompose REQUIRES theme, the seller PWA publish sequence is: POST /api/shops → POST /hero → PATCH /menu (edits, recomposes) → POST /compose. Reordering hero after the menu PATCH breaks publish for imported (theme-less) shops — encoded in seller/app.js, verified in scripts/e2e_smoke and an ad-hoc publish-sequence check.

5. Mock hero deliberately incompressible (and A5/A4 share one aspect ratio)

Mock mode renders gradient+noise (PIL effect_noise) instead of a flat fill so the downstream PDFs have realistic size (>10KB gate is meaningful; flat-color PNGs flate down to ~3KB PDFs). Cost: mock flyer PDFs are ~2.5MB — fine for dev, and real photos are similar. Gemini image generation only supports fixed ratios; A5/A4 (both ≈1:1.414 portrait) use "3:4" (nearest portrait), sticker "1:1"; the PDF draws full-bleed stretch, and the prompt's safe-zone constraint keeps the QR corner usable after the small distortion.

6. TTL cleanup scope

imagen.cleanup_expired() deletes only hero_*.png / hero_*.palette.json. Rehosted dish images (infra/media.py) and flyer PDFs have no TTL — dish images must survive source-CDN death (ARCH §4.5), and PDFs are cheap re-exports.

7. Added GET /api/shops/{slug}/orders (deviation: endpoint not in task list)

The Đơn tab needs a seller-facing order list to poll; OrderStore.list_by_shop existed with no HTTP surface. Added read-only endpoint, newest-first, limit param. Zero LLM, seller-side only.

8. Batch id embeds the folded location tag

create_batch("office plaza 1","a5") → office-plaza-1-a5-xxxx so buyer/context_rules.js's existing regex (office/văn phòng → office variant) classifies straight off the ?b= value with no server lookup, and analytics rows are human-readable.

9. Open question: whose account refunds a group split?

vietqr.group_split_payloads() fills the group-order placeholders from a passed account config. The e2e demo uses the SHOP's VietQR account, but ARCH §3.3 says members repay the payer (trưởng kèo) — a buyer whose personal bank account we never collect. MVP ships the helper + contract; the payer-account capture UX (ask closer for bank/STK at close time?) is an open product call for the group-order page phase.
